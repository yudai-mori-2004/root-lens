# Task 19: ランディングページの作成

## 目的

Title Protocolの公式ランディングページを作成する。
初見の一般人が「何を解決するのか」をパッと理解でき、読み進めるにつれ技術者が「なぜ信頼できるのか」を納得できる、誠実なページ。
最終的にGitHubリポジトリへの導線として機能し、OSSコミュニティへの参加を促す。

## コンテンツ設計方針

### 段階的開示（Progressive Disclosure）

ページは上から下に向かって抽象→具体へと展開する。どの深さで離脱しても、そこまでの情報だけで価値が伝わること。

```
[一般人が3秒で掴める]  ← ファーストビュー: 一文で何か分かる
[共感層]                ← 社会課題: 「あ、これ自分にも関係ある」
[納得層]                ← 仕組みの概要: 暗号もブロックチェーンも言わずに信頼構造を説明
[信頼層]                ← 技術詳細: C2PA, TEE, Solana — 専門家が検証できる情報
[行動層]                ← GitHub, ドキュメント, RootLensへの導線
```

### セクション構成

#### 1. ファーストビュー
- **一文**: デジタルコンテンツの「本物」を証明するオープンな基盤
- 「何ができるか」を視覚的に示す（例: 写真 → 証明付きリンク → 誰でも検証）
- 技術用語ゼロ。暗号もブロックチェーンもNFTも出さない
- CTAは「仕組みを見る」（スクロール誘導）と「GitHubで見る」の2つ

#### 2. 社会課題セクション — 「いま、何が起きているか」
- PROPOSAL.md §① の9領域の社会課題を網羅的に提示
- ただし全領域を等しく並べるのではなく、一般人に身近な順に配置:
  1. **SNS・日常**: 偽情報の拡散、AI生成コンテンツとの区別不能
  2. **クリエイター**: 無断転載、AI学習への無断利用
  3. **報道・メディア**: ディープフェイク動画、選挙への影響
  4. **金融・保険**: 証拠写真の偽造、保険詐欺
  5. **AI開発**: 学習データの権利問題、EU AI Act
  6. **司法・行政**: デジタル証拠の信頼性低下
  7. **医療**: 遠隔診療データの改ざん
  8. **学術・研究**: 論文画像の捏造、再現性危機
  9. **広告**: AI詐欺広告
- 各課題は事実ベース。PROPOSAL.mdに記載の具体的事例・数値を引用
- **絶対に誇張しない**。「〜と推計されている」「〜の事例がある」等、出典を明示

#### 3. 根本原因セクション — 「なぜ解決されていないのか」
- PROPOSAL.md §①の分析に基づく:
  - 事後検出（AI判定、電子透かし）はいたちごっこの構造
  - 暗号署名（C2PA）は「作成時に記録する」第二のアプローチで構造的優位
  - しかしC2PA単体では: SNSがメタデータ削除、検証結果を誰が保証するか不明、サービス間で分断
- 「検証結果を信頼するには、検証者を信頼しなければならない」という矛盾を提示

#### 4. Title Protocolの仕組み — 「どう解決するか」
- ここで初めて技術名を出すが、一般向けの説明を先行:
  - 「改ざんできない実行環境で検証」（TEE）
  - 「結果を書き換えられない台帳に記録」（ブロックチェーン）
  - 「誰でもノードを立てて検証に参加できる」（分散化）
- 図解: コンテンツ → TEE検証 → ブロックチェーン記録 → 誰でも確認可能
- PROPOSAL.md §②の4つの設計課題と解決策を簡潔に:
  - ステートレスTEEの二段階処理
  - E2E暗号化
  - Core/Extension アーキテクチャ
  - cNFTによるコスト最適化（月10万件で約2,250円）

#### 5. 既存サービスとの違い
- PROPOSAL.md §①の4つの例題をベースに:
  - 例1: メッセージアプリでの出自表明 → 既存はリンク送付のみ
  - 例2: SNSでの権利者追跡 → 既存はコンテンツ単体で出自を伝えられない
  - 例3: 拡散コンテンツの逆引き → 既存はサービス間で分断
  - 例4: 権利者への収益還元 → 既存は不可能
- 競合比較は攻撃的にならず、「何ができて何ができないか」の事実整理

#### 6. RootLens — 「最初のアプリケーション」
- Title Protocol上の最初の実装例としてRootLensを紹介
- 「撮る → 証明 → 共有」の3ステップ
- TP上に構築されることの意味（永続性・相互運用性・改ざん耐性）
- スクリーンショットまたはモック画像

#### 7. オープンソース & 参加への誘導
- GitHub リポジトリへのリンク
  - Title Protocol: https://github.com/yudai-mori-2004/title-protocol
  - RootLens: https://github.com/yudai-mori-2004/root-lens
- 仕様書（約3,000行）が公開されている事実
- 「誰でもノードを立てられる」「誰でもアプリを作れる」
- コントリビューション方法への導線

#### 8. フッター
- チーム紹介（簡潔に）
- 関連リンク（仕様書、GitHub、RootLensアプリ）
- ライセンス表記

### コンテンツ原則

1. **事実だけを書く**。推測や願望は書かない。数値には出典を添える
2. **「〜できる可能性がある」は使わない**。「〜ができる」か「〜はまだできない」
3. **技術用語は必要になるまで出さない**。出す時は必ず一般語での説明を先行させる
4. **競合を貶めない**。「Xにはできない」ではなく「TPはYを加えて実現する」
5. **100%、完全、絶対、などの断定表現を禁止**（仕様書 §3.1.2 準拠）
6. **日英両言語対応**（i18n）。初版は英語で作成（OSS国際コミュニティ向け）

## ビジュアルデザイン方針

### AI「平均デザイン」問題への対策

LLMは訓練データの統計的最頻パターンに収束する。以下のアンチパターンを明示的に回避し、
Title Protocolの「誠実さ」「公共基盤としての重厚さ」を反映したデザインを目指す。

#### 禁止事項（ネガティブ制約）

| やりがち | なぜダメか | 代わりに |
|----------|-----------|---------|
| Inter / Roboto / システムフォント | LLMデフォルト。個性がない | 目的に合ったフォントを選定（例: 見出しに重みのあるセリフ体、本文にヒューマニスト・サンセリフ） |
| Indigo / Purple グラデーション | Tailwindデフォルトの象徴。AI生成感が強い | ブランドカラーのネイビー(#1E3A5F)を軸に、TP固有のパレットを構築 |
| 3カラムFeatureカード | 最も統計的に頻出するレイアウト | 課題ごとに異なるレイアウトで変化をつける |
| 角丸カードのネスト | shadcn/ui的な均質感 | フラットなセクション分割、余白で区切る |
| ヒーローにグラデ背景 + 中央テキスト + CTA | 定番すぎて記憶に残らない | 内容に基づく構成を先に決め、それに合う表現を選ぶ |
| stock写真的なイラスト | 「それっぽいけど何も言ってない」 | 実際のプロダクトのスクリーンショット、技術図解、データ |
| Lucide / Heroicons一式 | AI生成サイトの共通言語 | 必要最小限のアイコン。装飾目的では使わない |
| bounce / elastic イージング | 2020年代前半感 | ease-out または cubic-bezier による落ち着いた動き |

#### デザインの方向性

- **誠実さ**: 装飾で信頼を演出しない。内容で信頼を得る。余白は多く、要素は少なく
- **重厚さ**: 公共基盤の重みを反映。軽薄なアニメーションは不要
- **読みやすさ**: 長文コンテンツが中心。タイポグラフィとリーディング体験を最重視
- **ダークモード**: ネイビー基調のダークテーマを基本とし、ライトモードも対応
- **レスポンシブ**: モバイルファーストだが、技術ドキュメントはデスクトップでの可読性も重視

#### 具体的なデザイン語彙（AIへの指示用）

以下の語彙を使ってデザインを指示する:

- **Typography**: Fluid type scale (clamp)、optical sizing、generous line-height (1.6-1.8 for body)
- **Color**: OKLCH color space、tinted neutrals (pure black/white禁止 — 暖かみのあるオフブラック/オフホワイト)
- **Space**: Vertical rhythm based on baseline grid、content-width max 680px for prose
- **Motion**: Reduced motion respected、scroll-triggered reveal (subtle opacity + translateY)、no decorative animation
- **Layout**: Single column prose sections、full-bleed for diagrams、asymmetric grid for comparison sections

## 技術実装方針

### スタック

- Next.js (既存webディレクトリ内に構築)
- 新規ルート: `/` をLPとして使用（既存の `/p/[pageId]` は公開ページとして維持）
- i18n: 既存の `next-intl` を活用（en/ja）
- CSS: 既存の `globals.css` + CSS Modules（Tailwind不使用 — 平均デザイン回避のため意図的に排除）
- フォント: `next/font` で最適化読み込み

### ページ構造

```
web/
  app/
    page.tsx           ← LP (現在の内容をリプレース)
    layout.tsx         ← 共通レイアウト
    p/[pageId]/        ← 既存: 公開ページ（変更なし）
  components/
    lp/                ← LP専用コンポーネント
      Hero.tsx
      SocialIssues.tsx
      RootCause.tsx
      HowItWorks.tsx
      Comparison.tsx
      RootLensIntro.tsx
      OpenSource.tsx
      Footer.tsx
  lib/
    lp-content.ts      ← テキストコンテンツ定義（i18n対応）
```

## 作業フロー

### Phase 0: コンテンツ・ファースト（デザインの前にテキストを完成させる）

平均デザインに陥る最大の原因は「テキストが決まる前にレイアウトを作る」こと。
コンテンツの構造と重みがデザインを決定する。逆ではない。

1. 全セクションのテキスト原稿を作成（英語）
2. 日本語訳を作成
3. テキストのみの状態でレビュー（Phase 3の品質チェックエージェントによる初回レビュー）

### Phase 1: デザイン（テキストに合わせてビジュアルを構築）

1. フォント選定: 見出し・本文・コード用の3書体を候補出しして決定
2. カラーパレット確定: ネイビー(#1E3A5F)を軸にOKLCHで展開
3. 各セクションのレイアウトをテキスト量と内容に基づいて個別設計
4. レスポンシブ対応

### Phase 2: 実装

1. コンポーネント実装（セクション単位）
2. i18n統合
3. OGP / メタデータ設定
4. パフォーマンス最適化（LCP, CLS）

### Phase 3: 品質チェック — ペルソナ・エージェント・レビュー

**最も重要なフェーズ。** LPの品質は「作る」ではなく「検証する」で決まる。

以下の5名のペルソナを持つレビューエージェントを用意し、それぞれの視点でLPを評価する。
各エージェントはページ全体を読んだ上で、そのペルソナの立場からフィードバックを返す。

#### エージェント一覧

| # | ペルソナ | 属性 | 評価観点 |
|---|---------|------|---------|
| A | **美容系インフルエンサー (25歳女性)** | Instagram 5万フォロワー。技術知識なし。AI加工疑惑を受けた経験あり | 「3秒で何か分かるか」「自分に関係あると思えるか」「難しくて離脱しないか」「使いたいと思うか」 |
| B | **フリーランス報道カメラマン (38歳男性)** | 紛争地域の取材経験あり。SNSで写真の真正性を疑われることに問題意識 | 「報道現場の課題が正しく記述されているか」「信頼できる印象か」「実際に使う導線があるか」 |
| C | **セキュリティエンジニア (32歳)** | 暗号技術・TEE・ブロックチェーンに詳しい。OSSコントリビューター | 「技術的な主張に嘘や誇張がないか」「TEEの信頼モデルを正直に説明しているか」「仕様書・コードへの導線は十分か」「コントリビュートしたくなるか」 |
| D | **大学3年生 (21歳)** | 情報工学専攻。ブロックチェーンに興味はあるが実務経験なし | 「技術的な説明が理解できるか」「学びになる内容があるか」「面白そうと思えるか」 |
| E | **損害保険会社の企画部門マネージャー (45歳)** | DX推進担当。保険金詐欺の被害額に問題意識。技術の詳細より実用性に関心 | 「業務課題の記述が的確か」「導入を検討する根拠が十分か」「誰に相談すればいいか分かるか」 |

#### レビュープロセス

1. 各エージェントにLP全文を読ませ、以下の形式でフィードバックを収集:
   - **理解度**: 0-10（どこまで理解できたか）
   - **信頼度**: 0-10（信頼できると感じたか）
   - **行動意欲**: 0-10（次のアクションを取りたいか）
   - **離脱ポイント**: どこで読むのをやめそうになったか
   - **疑問点**: 読後に残った疑問
   - **改善提案**: 具体的な変更提案

2. 全エージェントのフィードバックを集約し、以下の基準で判定:
   - **全員の理解度が7以上**: テキストは十分
   - **A（一般人）の理解度が5未満**: ファーストビュー〜社会課題セクションを改善
   - **C（技術者）の信頼度が7未満**: 技術セクションの正確性を再検証
   - **行動意欲が全員5以上**: 導線は機能している

3. 基準未達の場合、該当セクションを修正し、未達エージェントのみ再レビュー

### Phase 4: 最終確認

1. PROPOSAL.mdとの整合性チェック（事実関係の最終確認）
2. リンク動作確認（GitHub, RootLens, 仕様書）
3. OGP表示確認
4. Lighthouse スコア確認
5. 英語ネイティブチェック（不自然な表現の修正）
